# Fit Scoring

## Purpose

The fit score is not a magic truth number. It is a decision aid that helps the candidate decide whether to apply, skip, or position carefully.

## Score breakdown

```text
Must-have skill match:          30 points
Project evidence match:         25 points
Domain relevance:               15 points
Seniority match:                10 points
Tooling/platform match:         10 points
Resume positioning strength:     5 points
Risk check:                      5 points
Total:                         100 points
```

## Decision bands

```text
85–100: Strong Apply
70–84: Apply if interested
55–69: Maybe / careful positioning
Below 55: Skip
```

## How scoring should be interpreted

A high score means the profile and portfolio have enough evidence to create a credible application pack.

A lower score does not always mean the candidate is weak. It may mean the role expects a different track, such as research, deep platform ownership, or a tool stack not present in the portfolio.

## Examples

### Strong fit

A .NET + React + QA automation role can score well when the candidate has QA automation, API testing, Docker, CI/CD validation, and a financial automation project like Tradebot.

### Careful positioning

A role asking for Kubernetes can still be reasonable if the candidate has Docker and deployment validation exposure, but the resume must not claim deep Kubernetes production ownership without proof.

### Weak fit

A Senior ML Research Scientist role should score lower if the candidate has AI testing projects but no PhD-level research, publications, or model-training ownership.

## Hard rule

Fit scoring must explain itself. A raw number without reasoning is useless.
